# `wide_table_with_json` workload

This workload generates **write/update/delete** traffic against a pair of anonymized wide tables, with a focus on **large JSON-like payloads** and **compression cost behavior**.

It is useful for testing:

- Replication / CDC throughput with wide rows.
- Compression-heavy workloads, especially cases where zstd processing is expensive.
- Mixed update patterns (status-only, metadata updates, key updates, timestamp updates).

## Tables

For each shard index `n`, the workload uses two tables:

- Primary table: `wide_table_with_json_primary` / `wide_table_with_json_primary_<n>`
- Secondary table: `wide_table_with_json_secondary` / `wide_table_with_json_secondary_<n>`

`-table-count` controls how many shards to create, and `-table-start-index` controls the starting suffix.

## Payload sizing

`-row-size` controls the *approximate* payload width per row, and is split across columns:

- Primary: `payload_text` (varchar, clamped to 6144 bytes)
- Secondary: `payload_aux` (varchar, up to 3072 bytes), `payload_aux_2` (varchar, up to 3072 bytes), `payload_blob` (mediumblob, up to 16MB)

When `-row-size` is large, each column is clamped to its maximum.

## Payload generation

This workload always generates JSON-like payload records. The generated bytes are built from repeated structured fragments such as `{"key":"...","val":"...","ts":"...","extra":"..."}` so the data shape stays consistent with the workload name.

## Workload behavior (high level)

- Inserts are split between primary and secondary tables (roughly 77% / 23%).
- Updates are split between primary and secondary tables (roughly 51% / 49%).
- Primary updates include different access paths (by lookup key, by composite key, by primary id).
- Secondary updates include state-only updates and payload-heavy updates.
- Deletes target the primary table (by id or by composite key).

## Example

Create tables + run mixed writes with JSON-like payloads:

```bash
./workload -action write \
  -database-host 127.0.0.1 \
  -database-port 4000 \
  -database-db-name json_payload \
  -workload-type wide_table_with_json \
  -table-count 16 \
  -row-size $((16 * 1024)) \
  -thread 32 \
  -batch-size 32 \
  -percentage-for-update 0.5 \
  -percentage-for-delete 0.05
```

## Notes

- The DDL contains TiDB-specific TTL hints; other MySQL-compatible databases may ignore them.
- `-total-row-count` is used to derive an internal per-table update keyspace, so setting it closer to the actual prepared row count gives more realistic update hit rates.
